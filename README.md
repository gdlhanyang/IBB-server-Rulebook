## 0. 목적

본 가이드라인은 연구실 구성원이 Claude Code, ChatGPT, Copilot 등 AI 기반 코딩·분석 도구를 클러스터 환경에서 사용할 때 준수해야 할 기본 기준을 정리한 문서이다.

본 문서의 목적은 다음과 같다.

* 연구 데이터 및 개인정보 보호
* 공용 컴퓨팅 자원의 안정적이고 공정한 사용
* AI 활용 과정에서의 연구 윤리와 사용자 책임 명확화

AI 도구는 연구와 개발을 보조하는 수단이며, AI가 생성한 코드, 명령어, 분석 결과, 문서의 최종 책임은 사용자에게 있다.

---

## 1. 기본 원칙

### 1.1 사용자가 최종 검토한다

AI가 생성한 코드, 명령어, 분석 결과는 사용자가 직접 검토하고 검증한 뒤 실행하거나 연구 결과물에 사용한다.

### 1.2 이해하지 못한 명령은 실행하지 않는다

AI가 제안한 명령어의 목적과 영향을 이해하지 못한 경우 실행하지 않으며, 필요한 경우 담당자에게 문의한다.

### 1.3 되돌리기 어려운 작업은 별도로 확인한다

파일 삭제(`rm`), 권한 변경(`chmod`, `chown`), 데이터 덮어쓰기, 대규모 파일 이동 등은 실행 전 사용자가 직접 재확인한다.

### 1.4 공용 자원을 우선적으로 고려한다

클러스터는 공동으로 사용하는 연구 인프라이므로, 개인 작업의 편의보다 전체 사용자의 접근성과 안정성을 우선한다.

## 2. 데이터 보안 및 입력 기준

AI 도구는 사용자가 입력한 프롬프트, 코드, 파일 내용, 로그 또는 명령어 일부를 외부 서버로 전송할 수 있다. 따라서 외부 AI 도구에 입력하는 정보는 사전에 보안 수준을 판단해야 한다.

### 2.1 외부 AI 도구 입력 금지 정보

다음 정보는 외부 AI 도구에 입력하지 않는다.

* 환자 또는 임상 데이터
* 개인식별정보(PII)
* 미공개 시퀀싱 데이터 및 민감 연구 데이터
* FASTQ, BAM, CRAM, VCF, BCF 등 human raw sequencing 또는 variant 원자료
* 로그인 정보, 비밀번호, API key, SSH key, token
* 공동연구 비밀유지계약(NDA) 대상 원자료
* 외부 공개가 제한된 미발표 논문의 핵심 데이터 또는 결과

### 2.2 데이터 등급 및 사용 기준

| 등급                    | 예시                                                                                                    | 외부 AI 도구 입력 기준                                   |
| --------------------- | ----------------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| Public                | 공개 논문, 공개 GitHub 코드, 공개 reference genome 정보, 공개 튜토리얼                                                  | 사용 가능                                            |
| Internal / Restricted | 연구실 내부 코드, 미공개 분석 흐름, 공동연구 관련 비공개 정보, 미발표 결과 일부, 내부 서버 사용 정보                                          | PI 승인 후, 민감 정보와 실제 데이터는 제거하거나 마스킹한 뒤 제한적으로 사용 가능 |
| Prohibited            | 환자정보, 개인식별정보, human raw sequencing data, FASTQ/BAM/VCF 원자료, SSH key, API key, token, 비밀번호, NDA 대상 원자료 | 사용 금지                                            |

판단이 어려운 경우 사용자는 임의로 AI 도구에 입력하지 않고 PI 또는 지정된 담당자에게 문의한다.

### 2.3 사용 가능 범위 예시

AI 도구 사용 가능 범위는 다음과 같이 구분한다.

| 구분            | 예시                                                                                                         | 사용 기준                                                   |
| ------------- | ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------------- |
| 허용            | 일반적인 Python/R/bash 코드 디버깅, Slurm script 초안 작성, 에러 메시지 해석, 공개 데이터 기반 분석 코드 개선                               | 민감 정보나 미공개 데이터가 포함되지 않는 경우 사용 가능                        |
| PI 승인 후 사용 가능 | 연구실 내부 코드, 미공개 프로젝트의 분석 흐름, 공동연구 관련 분석 코드, 내부 서버 사용 방식, 미발표 결과 일부                                          | 실제 데이터, 개인정보, 공동연구 기밀이 포함되지 않도록 마스킹한 뒤 PI 승인 후 제한적으로 사용 |
| 금지            | 환자/임상 데이터, 개인식별정보, human raw sequencing data, FASTQ/BAM/VCF 원자료, API key, SSH key, token, 비밀번호, NDA 대상 원자료 | 외부 AI 도구에 입력 금지                                         |

### 2.4 입력 정보의 일반화 및 마스킹

AI 도구에 질문할 때는 가능한 한 실제 연구 데이터가 아닌 예시 데이터를 사용한다. 파일 경로, 서버 주소, 계정명, 샘플명, 공동연구자명 등 내부 정보는 필요 최소한으로만 포함하고, 가능한 경우 일반화하거나 마스킹한다.

## 3. 컴퓨팅 자원 사용 규칙

### 3.1 마스터 노드(`ibb-master`) 사용 제한

마스터 노드는 로그인, 파일 편집, 작업 제출, 가벼운 조회를 위한 용도로 사용한다. 다음 작업은 마스터 노드에서 수행하지 않는다.

* 연산 또는 분석 작업 실행
* 대규모 파일 검색, indexing, parsing, compression/decompression
* AI agent에 의한 광범위한 코드베이스 검색 또는 대량 파일 처리
* 장시간 실행되는 background process, watcher, web server 실행

AI 코딩 도구는 내부적으로 `rg` 등 파일 검색 프로세스를 실행할 수 있으므로, 작업 범위를 프로젝트 단위로 제한하고 필요한 경우 compute node에서 사용한다.

### 3.2 연산 작업은 compute node에서 수행

30초 이상 소요될 가능성이 있는 작업은 Slurm을 통해 compute node에 제출한다. 일반적인 배치 작업은 `sbatch`를 사용하고, 짧은 대화형 작업에 한해 `srun --pty` 또는 `salloc`을 사용한다.

### 3.3 대화형 세션 사용 기준

대화형 세션 사용 시 다음 사항을 준수한다.

* 작업이 끝난 세션은 즉시 종료한다(`exit`).
* 코어, 메모리, GPU 등 자원을 실제 필요량보다 과도하게 예약하지 않는다.
* 사용하지 않는 상태로 노드를 장시간 점유하지 않는다.
* 자원 예약 대비 실사용이 낮은 상태가 지속되지 않도록 한다.

### 3.4 AI가 작성한 Slurm script 제출 전 확인

AI가 작성한 Slurm script는 제출 전 사용자가 직접 검토한다. 특히 다음 항목을 확인한다.

| 확인 항목               | 기준                            |
| ------------------- | ----------------------------- |
| `--time`            | 작업 예상 시간에 비해 과도하지 않은지 확인      |
| `--mem`             | 실제 필요량보다 지나치게 크게 요청하지 않았는지 확인 |
| `--cpus-per-task`   | 작업 특성에 맞는 CPU core 수인지 확인     |
| `--gres`            | GPU 요청이 반드시 필요한 작업인지 확인       |
| Output/error log 경로 | 로그가 적절한 위치에 저장되는지 확인          |
| Array job 규모        | 과도하게 많은 작업을 생성하지 않는지 확인       |
| 자동 재제출 또는 반복 실행     | 의도하지 않은 반복 제출이 없는지 확인         |
| 무한 루프 또는 장시간 유휴 점유  | 노드 장시간 점유 가능성이 없는지 확인         |

무한 루프, 자동 재제출, 과도한 array job, GPU 장시간 점유는 금지한다. AI가 생성한 분석 코드나 shell script는 전체 데이터에 적용하기 전 작은 테스트 데이터 또는 일부 샘플에서 먼저 검증한다.

### 3.5 자원 사용 확인 및 종료

사용자는 본인 작업의 상태와 자원 사용량을 주기적으로 확인한다.

```bash
squeue -u $USER
scontrol show node <노드명>
scancel <JobID>
```

불필요한 작업이나 잘못 제출된 작업은 즉시 종료한다.

## 4. AI 도구 사용 시 안전 수칙

### 4.1 권한 모드는 보수적으로 설정한다

자동 실행(auto-approve)보다 명령 확인 모드 또는 read/search 중심 모드를 기본으로 사용한다.

### 4.2 작업 범위를 제한한다

AI가 접근하거나 수정할 수 있는 범위는 현재 프로젝트 폴더로 제한한다. 홈 디렉토리 전체, 공용 폴더 전체, 대용량 데이터 저장소 전체를 대상으로 하지 않는다.

### 4.3 공용 디렉토리는 자동 수정 대상에서 제외한다

`/home/shared`, `/data`, `/project`, `/scratch` 등 공용 디렉토리는 AI agent가 자동으로 수정하지 않도록 한다. 변경이 필요한 경우 사용자가 직접 확인한다.

### 4.4 위험 명령은 자동 승인하지 않는다

다음 명령 또는 이에 준하는 작업은 AI agent의 자동 승인 대상에서 제외한다.

| 구분            | 예시                                                   |
| ------------- | ---------------------------------------------------- |
| 파일 삭제         | `rm`, `find ... -delete`                             |
| 파일 이동 또는 덮어쓰기 | `mv`, `cp -f`, `rsync --delete`                      |
| 권한 변경         | `chmod`, `chown`                                     |
| 작업 제출 또는 취소   | `sbatch`, `scancel`                                  |
| 외부 스크립트 실행    | `curl \| bash`, `wget \| bash`                       |
| 인증 관련 파일 수정   | `.ssh/`, `.bashrc`, `.condarc`, `.gitconfig`, `.env` |

### 4.5 버전 관리를 사용한다

중요한 코드는 `git` 등으로 변경 이력을 남긴 뒤 AI 수정 작업을 진행한다.

### 4.6 인증 정보는 분리하여 관리한다

AI 도구 로그인 정보, token, key는 본인 계정에 한정하여 사용하며 타인과 공유하지 않는다.

### 4.7 도구 설정을 점검한다

AI 도구의 기본 환경 설정, 활성화된 plugin 또는 extension, system prompt, 자동 실행 권한이 본 가이드라인에 부합하는지 확인한다.

## 5. 연구 윤리 및 책임

* AI가 생성한 코드, 문장, 분석을 연구 결과물에 사용할 경우 소속 기관, 학술지, 학회의 AI 사용 고지 규정을 따른다.
* AI가 생성한 결과의 정확성, 재현성, 적절성은 사용자가 검증한다.
* AI를 표절, 데이터 위변조, 부적절한 저자 기여 표시 등 연구 부정행위에 사용하지 않는다.
* 연구 분석, 코드 작성, 문서 작성에 AI 도구를 사용한 경우 필요에 따라 도구명, 사용 날짜, 사용 목적, AI가 생성 또는 수정한 범위를 기록한다.

## 6. 보안 사고 대응 절차

API key, SSH key, token, 비밀번호, 환자정보, 개인식별정보, 미공개 연구 데이터 등을 AI 도구에 입력했거나 입력했을 가능성이 있는 경우 사용자는 즉시 다음 절차를 따른다.

1. 해당 AI 세션 사용을 중단한다.
2. 노출 가능성이 있는 key, token, password를 폐기하고 재발급한다.
3. PI 또는 지정된 담당자에게 보고한다.
4. 발생 시각, 사용한 도구, 입력한 정보의 종류, 노출 가능 범위를 기록한다.
5. 필요한 경우 기관 보안 담당자, 공동연구자, 데이터 제공기관과 상의하여 후속 조치를 진행한다.

보안 사고 가능성이 있는 경우 사용자는 임의로 판단하거나 숨기지 않고 신속히 보고한다. 신속한 보고는 추가 피해를 방지하기 위한 절차이다.

## 9. 개정 메모: 하유진 추가사항

v0.3에서는 기존 초안의 형식을 유지하되, 다음 사항을 보완하였다.

* 데이터 등급을 Public, Internal / Restricted, Prohibited로 구분하고, Internal / Restricted 정보는 PI 승인 후 제한적으로 사용할 수 있도록 정리하였다.
* AI 도구 사용 사례를 허용, PI 승인 후 사용 가능, 금지로 구분하였다.
* AI가 작성한 Slurm script 제출 전 `--time`, `--mem`, `--cpus-per-task`, `--gres`, output/error log 경로, array job 규모, GPU 요청 필요성 등을 확인하도록 명시하였다.
* 무한 루프, 자동 재제출, 과도한 array job, GPU 장시간 점유를 금지 사항으로 추가하였다.
* AI agent의 자동 실행 권한을 보수적으로 설정하고, 삭제, 덮어쓰기, 권한 변경, 작업 제출/취소, 외부 스크립트 실행 등 위험 명령은 자동 승인하지 않도록 구체화하였다.
* Key, token, password, 환자정보, 개인식별정보, 미공개 연구 데이터가 AI 도구에 입력되었거나 입력되었을 가능성이 있는 경우의 보안 사고 대응 절차를 추가하였다.
* 문서 전반에서 이모지와 구어체 표현을 제거하고, 내부 규정 문서에 맞는 간결한 문체로 정리하였다.
